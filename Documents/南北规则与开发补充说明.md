# 南北规则与开发补充说明

## 一、补充目的
为了让项目可在 5 周内落地，同时满足“规则切换 + AI策略 + 联机扩展”的课程要求，本说明补充：
1. 当前代码中南北规则的可执行定义
2. 后续迭代中建议优先级
3. 团队执行时的对齐标准

## 二、南北规则在当前代码中的定义

### 1. 南方规则（`RuleSetType.SOUTH`）
1. 首轮出牌必须包含方块 3。
2. 点数相同可继续比较花色。
3. 顺子允许 `A2345`。
4. 首轮先手优先由方块 3 持有者开始。

### 2. 北方规则（`RuleSetType.NORTH`）
1. 首轮不强制方块 3。
2. 点数相同不比较花色。
3. 顺子不允许 `A2345`。
4. 若有上局胜者，则由上局胜者先手，否则默认 1 号位先手。

## 四、测试覆盖点
1. 牌型识别：同花顺识别、`A2345` 在南北规则下差异。
2. 出牌限制：南方首轮方块 3 约束，北方首轮放宽。
3. 比较规则：同点数下南方可按花色继续比较。
4. 结算逻辑：输家倍率计分与赢家累计。

## 五、后续建议优先级
1. P0：把蓝牙同步层接入 `GameController`，打通房间到对局。
2. P0：UI 对局页面接入真实选牌与回合渲染。
3. P1：完善五张牌型边界测试（同花五与顺子冲突、铁支比较等）。
4. P1：增加“顶大”规则（只剩一张需报牌）与提醒机制。
5. P2：扩展 AI（保牌、拆牌、终局搜索）并做策略对比实验。

## 六、风险与注意事项
1. 不同资料对“北方规则”定义不完全一致，本项目当前为“课程可交付版本”的工程化定义。
2. 最终答辩前应由组长统一规则口径，并在报告中明确版本说明。
3. 蓝牙层目前是协议占位，未包含完整连接状态机，需要第 3 周集中攻关。
